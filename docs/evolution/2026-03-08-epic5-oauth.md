## Summary

Social login system with Google/GitHub OAuth and cookie-based reader sessions. 12 acceptance scenarios across the OAuth feature file, 5 TDD steps, all GREEN. Introduces the Identity vertical slice with ReaderSession entity, IOAuthClient port, and session cookie management.

## Architecture

### Key Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Session model | Cookie-based ReaderSession (server-side) | Simple, no client-side token management; works with Astro SSG |
| OAuth abstraction | IOAuthClient port with provider-specific adapters | Hexagonal boundary; test with NSubstitute, swap providers without touching use cases |
| Dual auth model | JWT (admin) + Session cookie (reader) | Admin needs stateless auth for API clients; readers need simple browser-based sessions |
| Session expiry | 30-day sliding window | Balance between convenience and security; configurable via IClock |
| AuthProvider enum | Google, GitHub | Two most common developer OAuth providers; extensible via enum |

### New Components

- **Domain**: `ReaderSession` entity (session ID, display name, avatar, provider, expiry), `AuthProvider` enum
- **Application**: `HandleOAuthCallback`, `CheckSession`, `InitiateOAuth`, `SignOut` use cases
- **Application ports**: `IOAuthClient` (driven port for OAuth provider interaction), `IReaderSessionRepository`
- **Infrastructure**: `ReaderSessionConfiguration` (EF Core), `EfReaderSessionRepository`, `AddReaderSessionsTable` migration
- **API**: `OAuthEndpoints` — GET `/api/oauth/{provider}/initiate`, GET `/api/oauth/callback`, GET `/api/oauth/session`, POST `/api/oauth/signout`
- **Test infrastructure**: `OAuthApiDriver`, `OAuthSteps`, `ReaderSessionContext` (shared state for session cookie)

## Lessons Learned

1. **IOAuthClient port is the hexagonal boundary** — All OAuth provider complexity (token exchange, user info fetching) lives behind a single port. Acceptance tests use NSubstitute to simulate provider responses, making tests fast and deterministic.
2. **Session cookie in acceptance tests requires shared context** — `ReaderSessionContext` holds the session cookie across steps, allowing "Given a reader session exists" to flow into "When the reader posts a comment" seamlessly.
3. **Dual auth is clean when separated at the endpoint level** — `RequireAuthorization()` for admin, `AllowAnonymous()` + use-case-level session validation for readers. No middleware confusion.
