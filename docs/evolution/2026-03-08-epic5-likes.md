## Summary

Anonymous like/unlike system with visitor-based deduplication. 14 acceptance scenarios across 2 feature files (WalkingSkeleton + Likes), 5 TDD steps, all GREEN. Introduces the Engagement vertical slice with Like entity, VisitorId value object, and dedicated repository port.

## Architecture

### Key Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Visitor identity | Cookie-based `VisitorId` (UUID) | Anonymous likes without requiring authentication; simple, privacy-friendly |
| Like deduplication | Composite PK (PostSlug + VisitorId) | Database-enforced uniqueness; idempotent like/unlike operations |
| Like as independent aggregate | References BlogPost by Slug, not FK to ID | Keeps engagement decoupled from content lifecycle; avoids aggregate contention |
| Idempotent unlike | Return success even if not liked | Client-friendly; no need to check state before unlike |

### New Components

- **Domain**: `VisitorId` value object (validated UUID), `Like` entity (composite identity: PostSlug + VisitorId)
- **Application**: `LikePost`, `UnlikePost`, `GetLikeCount`, `CheckIfLiked` use cases with Result pattern
- **Infrastructure**: `LikeConfiguration` (EF Core, composite PK, snake_case), `EfLikeRepository`, `AddLikesTable` migration
- **API**: `LikeEndpoints` — POST/DELETE `/api/posts/{slug}/likes`, GET `/api/posts/{slug}/likes/count`, GET `/api/posts/{slug}/likes/check`
- **Test infrastructure**: `LikeApiDriver`, `LikeSteps` step definitions, `WalkingSkeleton.feature`

## Lessons Learned

1. **Composite PK = free deduplication** — The (PostSlug, VisitorId) composite primary key means the database enforces "one like per visitor per post" without any application-level checks. Idempotent by design.
2. **Value Objects absorbed validation complexity** — `VisitorId` rejecting empty GUIDs and `Like` enforcing its invariants meant error path scenarios passed with zero additional production code.
3. **Walking skeleton proves the vertical slice** — The first 2 scenarios exercised every layer (domain → application → infrastructure → API → acceptance test). Every subsequent step only added behavior, never infrastructure.
